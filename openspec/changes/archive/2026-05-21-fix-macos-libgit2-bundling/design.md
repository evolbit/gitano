## Context

The macOS crash report shows dyld aborting before Gitano reaches application code:

- missing library: `/opt/homebrew/*/libgit2.1.9.dylib`
- referenced by: `/Applications/gitano.app/Contents/MacOS/gitano`
- termination namespace: `DYLD`, code `1`, `Library missing`

This means the release binary was linked against a developer-machine Homebrew copy of `libgit2`. The project currently depends on `git2 = "0.20.2"` in `src-tauri/Cargo.toml`; that pulls `libgit2-sys 0.18.2+1.9.1`, whose default build path can discover system/package-manager libraries through `pkg-config`. That is fine for local development, but it is not acceptable for a distributed `.app`.

The fix should make the native Git stack portable at build time and add a packaging check that catches the exact failure before release.

## Goals / Non-Goals

**Goals:**
- Make macOS `.app` bundles launch on machines without Homebrew, MacPorts, or a manually installed `libgit2`.
- Prefer a build-time dependency strategy that embeds or statically links the native Git library instead of copying a Homebrew dylib into the app.
- Verify the packaged executable does not reference package-manager paths for `libgit2`.
- Keep existing Git behavior unchanged.
- Document the validation so future `git2` / `libgit2-sys` upgrades remain safe.

**Non-Goals:**
- Changing Git feature behavior, repository operations, or UI flows.
- Replacing `git2` with shelling out to the system `git` binary.
- Building a full notarization/signing pipeline.
- Solving unrelated native dependencies unless the validation exposes the same package-manager path problem.

## Decisions

### Use vendored native Git dependencies for release portability

Configure the Rust `git2` dependency so the release build does not resolve `libgit2` from `pkg-config` or Homebrew. The preferred path is enabling `git2`'s `vendored-libgit2` feature. Because `git2` defaults include HTTPS support through OpenSSL, implementation should also evaluate `vendored-openssl` and use it if `otool` shows package-manager OpenSSL references.

This keeps the Git integration inside the existing Rust API and avoids a runtime installer requirement.

Alternative considered: copy Homebrew's `libgit2.1.9.dylib` into `Contents/Frameworks` and rewrite install names. That can work, but it couples the app to a developer-machine package layout and makes transitive dependency handling fragile.

Alternative considered: require users to install Homebrew `libgit2`. That is unacceptable for a desktop app because the app crashes before it can explain the requirement.

### Add a macOS bundle dependency verification script

Add a small release verification script that inspects the packaged executable with `otool -L`. The check should fail when:

- any `libgit2` install name resolves to `/opt/homebrew`, `/usr/local`, or another package-manager prefix
- any `libgit2` install name is not a system library, an executable-relative path, or absent because it was statically linked
- the expected `.app` executable cannot be found after a bundle build

The script should print the relevant `otool -L` output on failure so the problem is immediately diagnosable.

### Keep runtime code unchanged

No frontend or backend Git command behavior should change. The problem is at dynamic linking and packaging time, so the implementation should stay in dependency configuration and release verification unless validation uncovers a specific transitive native library that must be bundled.

### Verify the packaged artifact, not only `cargo check`

`cargo check` can pass while the release `.app` still has an invalid dynamic library install name. Verification must run against the actual Tauri macOS bundle executable generated by a release build.

## Risks / Trade-offs

- [Vendored native builds increase build time] -> Accept the cost for release reliability; keep the change limited to the Git native stack.
- [Vendored OpenSSL can increase binary size] -> Only include it if needed to remove non-portable package-manager references from the shipped executable.
- [Different macOS architectures may resolve dependencies differently] -> Validate the built artifact for the target architecture and document the command used.
- [CI may not run on macOS] -> Keep the verification script local and CI-friendly, then wire it into macOS release jobs when available.
- [Static linking can hide the expected `libgit2` line from `otool`] -> Treat absence of a `libgit2` dylib reference as success.

## Migration Plan

1. Update `src-tauri/Cargo.toml` so `git2` uses vendored native dependency features needed for a portable macOS bundle.
2. Refresh `src-tauri/Cargo.lock`.
3. Add a macOS bundle dependency verification script.
4. Build the app bundle on macOS and run the verification script against the packaged executable.
5. Run existing Rust checks/tests to confirm Git behavior still compiles.

Rollback is straightforward: revert the manifest feature change and verification script. That restores the previous build behavior, but would also restore the launch-crash risk.
